# Технический бриф для Codex: развитие память.site

**Версия:** 1.0  
**Дата:** 18 июля 2026 года  
**Продукт:** память.site — интерактивная карта памятных мест Тюмени  
**Назначение документа:** пошаговая реализация новых функций в существующем проекте без переписывания приложения и без регрессий текущего функционала.

---

## 1. Главная инструкция Codex

Этот документ описывает большую программу развития продукта. **Нельзя реализовывать весь бриф одним PR, одним коммитом или одним непрерывным рефакторингом.**

Codex должен:

1. Сначала выполнить только **Фазу 0 — аудит существующего проекта**.
2. Изучить `AGENTS.md`, `README`, документацию, структуру репозитория, переменные окружения, схему БД, миграции, тесты и процесс деплоя.
3. Не предполагать конкретный фреймворк, ORM, карту, хранилище или систему авторизации до завершения аудита.
4. Сохранить текущий стек и архитектурный стиль проекта, если в аудите не обнаружена критическая причина для локального изменения.
5. После аудита сопоставить предложенные ниже сущности и API с фактическими таблицами, моделями, маршрутами и соглашениями проекта.
6. Реализовывать **одну фазу за раз**, разбивая её на небольшие независимые задачи.
7. Перед каждой фазой представить план файлов, миграций, API, тестов, рисков и отката.
8. Не переходить к следующей фазе, пока не выполнены критерии приёмки текущей.
9. Не деплоить в production без отдельного прямого указания владельца проекта.
10. Не менять конфигурацию чужих сервисов на сервере, не переиспользовать их порты, процессы, БД, каталоги или секреты.

Если требования брифа конфликтуют с уже работающей реализацией, Codex обязан:

- сохранить обратную совместимость;
- описать конфликт;
- предложить минимальное изменение;
- дождаться решения владельца только в случае, если выбор существенно влияет на продукт или данные.

## 2. Текущее состояние, которое нельзя сломать

По состоянию публичной версии память.site в продукте уже есть:

- интерактивная карта памятных мест Тюмени;
- около 92 объектов;
- поиск по карте;
- категории объектов;
- фильтрация и отображение количества объектов;
- переключение «Карта / Список / Мероприятия»;
- режим «Рядом со мной»;
- избранные и посещённые места;
- карточка объекта с адресом и округом;
- фотографии, видео и 3D-модели;
- аудиогид;
- текстовая расшифровка аудиогида;
- структурированные разделы статьи;
- источники;
- сообщение об ошибке;
- переход к маршруту в Яндекс Картах;
- публикация ссылки на объект;
- раздел мероприятий с поиском и фильтром по периоду;

- планируемая или частично реализованная серверная генерация аудио через Yandex SpeechKit.

Все существующие публичные URL, идентификаторы объектов, избранное, посещения, медиа, тексты, аудио и SEO-данные должны сохраняться.

## 3. Целевое позиционирование продукта

память.site должен развиваться из справочной карты в платформу сохранения городской памяти со сквозным сценарием:

> узнать об объекте → пройти маршрут → найти людей и события → дополнить историю → помочь сохранить объект

Продукт должен поддерживать три направления:

1. **Публичный городской гид:** карта, маршруты, аудиопрогулки, архивные материалы и доступные сценарии.
2. **Общественный архив:** модерируемые семейные истории, фотографии, документы и голосовые воспоминания.
3. **B2B/B2G-система:** цифровой реестр, состояние объектов, QR/NFC, аналитика и кабинеты организаций.

## 4. Принципы реализации

### 4.1. Обратная совместимость

- Не переименовывать существующие таблицы, URL и поля без миграционного слоя.
- Миграции сначала должны быть расширяющими: новые nullable-поля и таблицы, затем backfill, затем ограничения.
- Существующие опубликованные объекты при добавлении workflow должны получить статус `published`.
- Старые ссылки должны продолжить работать либо иметь постоянный redirect.
- Не удалять старые поля до завершения переноса данных и отдельного подтверждения.

### 4.2. Feature flags

Каждая крупная функция должна включаться независимо:

- `routes_enabled`;
- `route_gps_autoplay_enabled`;
- `offline_packages_enabled`;
- `knowledge_graph_enabled`;
- `timeline_enabled`;
- `qr_campaigns_enabled`;
- `condition_registry_enabled`;
- `community_archive_enabled`;
- `education_enabled`;
- `multilingual_enabled`;
- `ai_guide_enabled`;
- `partner_portal_enabled`;
- `ar_enabled`.

Использовать существующий механизм конфигурации. Не добавлять отдельную тяжёлую систему feature flags, если достаточно текущих env/config/DB-настроек.

### 4.3. Источники и достоверность

- Исторические факты публикуются только после редакционной проверки.
- Каждый значимый факт должен иметь источник или явную пометку редактора.
- ИИ не публикует тексты, переводы и ответы автоматически.
- Внешние ссылки на архивы хранятся как ссылки и метаданные; не копировать массово внешние данные без отдельного решения по правам и интеграции.

### 4.4. Privacy by design

- Не хранить точную геолокацию посетителя без отдельного согласия.
- Для аналитики использовать псевдонимный идентификатор сессии.
- Не хранить полный IP дольше технически необходимого; для отчётов использовать обезличенные данные.
- Контакты автора пользовательского материала хранить отдельно от публичной атрибуции.
- Удалять EXIF из публичных копий файлов по умолчанию.
- Не собирать персональные данные школьников; использовать временный код класса и псевдонимную сессию.
- Перед публичным запуском народного архива и кабинетов провести отдельную юридическую проверку текстов согласий, политики и сроков хранения.

### 4.5. Mobile-first и доступность

- Основной сценарий — смартфон на улице.
- Все новые функции должны работать без установки нативного приложения.
- Цель по доступности — WCAG 2.2 AA, но публично заявлять соответствие можно только после аудита.
- Любая функция GPS, камеры, уведомлений или офлайна должна иметь понятный ручной fallback.

### 4.6. Общие UX-инварианты

- Объекты, маршруты, люди, события и опубликованные архивные материалы имеют стабильные прямые URL.
- Обновление страницы и открытие ссылки в новой вкладке восстанавливают тот же публичный экран.
- Кнопки браузера «Назад» и «Вперёд» корректно возвращают карту, список, карточку и маршрут.
- Базовый просмотр, QR-переход, аудио и школьная сессия не требуют обязательной регистрации.
- Для каждого экрана предусмотрены `loading`, `empty`, `error`, `offline` и `permission denied` состояния по необходимости.
- Проверять интерфейс минимум на ширинах 360, 768 и 1440 CSS px.
- Состояние не кодируется только цветом; у каждого статуса есть текст.
- Карта никогда не является единственным способом получить список объектов и действий.
- Новый глобальный аудиоплеер, если он потребуется, не должен создавать второй независимый плеер рядом с существующим: текущий компонент следует расширить или переиспользовать.

## 5. Роли и права

Codex должен сопоставить роли с текущей системой авторизации и не создавать дубликаты.

| Роль | Основные права |
|---|---|
| Гость | Просмотр карты, объектов, маршрутов, аудио, событий и публичных архивных материалов |
| Пользователь | Синхронизация избранного, посещений, маршрутов и отправка материалов; авторизация не должна быть обязательна для базового просмотра |
| Автор | Создание и редактирование черновиков контента |
| Редактор | Проверка фактов, источников, переводов и аудиосценариев |
| Модератор | Работа с пользовательскими материалами и жалобами |
| Инспектор | Осмотры состояния, дефекты, фото и статусы работ |
| Преподаватель | Создание учебных сессий и просмотр агрегированных результатов |
| Аналитик | Просмотр обезличенной аналитики организации |
| Администратор организации | Пользователи и материалы одной организации |
| Системный администратор | Глобальные настройки и полный аудит действий |

Права проверяются на сервере. Скрытие кнопки в интерфейсе не является контролем доступа.

## 6. Предлагаемая модель данных

Названия ниже являются логическими. Codex должен адаптировать их к текущему naming style и не создавать сущность, если эквивалент уже существует.

### 6.1. Контент и редакционный процесс

| Сущность | Ключевые поля |
|---|---|
| `content_sources` | type, title, author, publisher, year, url, archive_code, access_date, verification_status |
| `content_versions` | entity_type, entity_id, version, payload/diff, author_id, created_at, reason |
| `editorial_tasks` | entity_type, entity_id, status, assignee_id, reviewer_id, deadline, comments |
| `audit_log` | actor_id, organization_id, action, entity_type, entity_id, before, after, created_at |

Статусы контента:

`draft → review → changes_requested → approved → published → archived`

### 6.2. Расширенный паспорт места

Существующую сущность объекта не заменять. При необходимости расширить её или добавить связанную таблицу:

- альтернативные названия;
- тип объекта;
- дата/период создания;
- дата открытия;
- авторы и организации;
- охранный статус;
- материалы;
- координаты и адрес;
- режим доступа;
- атрибуты доступности;
- источники;
- права на медиа;
- дата последней проверки;
- дата последнего обновления карточки;
- статус достоверности.

### 6.3. Аудиоверсии

`audio_variants`:

- `place_id` или другой владелец контента;
- `variant`: `short`, `full`, `child`, `easy_language`, `audio_description`, `route_intro`, `route_direction`;
- `locale`;
- `script_text`;
- `voice`;
- `speed`;
- `status`: `empty`, `stale`, `queued`, `processing`, `ready`, `failed`;
- `audio_url`/storage key;
- `text_hash`;
- `version`;
- `duration_seconds`;
- `generated_at`;
- `error_code` и безопасное сообщение об ошибке;
- `generation_provider`;
- `manual_upload`;
- `approved_by` и `approved_at`.

Если в текущем коде уже используются статусы `draft/processing/ready/error`, выполнить явное сопоставление вместо создания второго независимого workflow.

### 6.4. Маршруты

`routes`:

- slug, status, title, summary, description, cover;
- theme/category;
- mode: walking/bicycle/car;
- estimated duration, distance, audio duration;
- difficulty, season, recommended time;
- age group;
- accessibility profile;
- start/end coordinates;
- geometry/polyline при наличии;
- organization/author;
- locale and translation status;
- offline package version;
- publication dates.

`route_stops`:

- route_id;
- place_id;
- position;
- arrival radius;
- recommended stop duration;
- route-specific intro and directions;
- short/full audio variant;
- GPS autoplay flag;
- optional quiz/task;
- optional alternative accessible stop.

`route_progress` создаётся только для авторизованного пользователя. Для гостя прогресс хранится локально и переносится в аккаунт только после явного входа.

### 6.5. Люди, события и связи

Предпочтительно использовать типизированные сущности, а не одну бесконтрольную polymorphic-таблицу:

- `people` — имя, варианты имени, годы жизни, краткая и полная биография, portrait, verification status;
- `historical_events` — название, дата/диапазон, описание, география;
- `organizations` — учреждения, подразделения, авторы и ответственные организации;
- `archive_documents` — название, вид документа, дата, архивный шифр, ссылка, права;
- `place_people` — place_id, person_id, role/relation_type;
- `place_events` — place_id, event_id, relation_type;
- `person_events` — person_id, event_id, relation_type;
- `entity_sources` — источник для конкретного утверждения или сущности.

Нужны alias-поиск и нормализация ФИО. Внешние ссылки на ОБД «Мемориал», «Память народа» и другие архивы должны иметь тип, URL и дату проверки.

### 6.6. Хронология и «тогда/сейчас»

`timeline_entries`:

- place_id;
- type: creation/opening/move/damage/restoration/commemoration/other;
- точная дата, год либо диапазон;
- title, description;
- media;
- source IDs;
- public status and editorial status.

`archive_media`:

- place_id;
- capture date/year range;
- source and rights;
- original author;
- approximate camera coordinates and direction при наличии;
- paired current media ID;
- public file derivatives;
- alt text;
- verification status.

### 6.7. QR, NFC и короткие ссылки

`short_links`:

- случайный стабильный code;
- internal target type and ID;
- locale;
- enabled status;
- optional expiration;
- redirect rules without arbitrary external destination.

`qr_campaigns`:

- short_link_id;
- placement name/type;
- organization;
- object/route;
- print batch;
- active dates;
- campaign tags;
- scan counters/analytics reference.

Один физический QR/NFC должен продолжать работать после изменения карточки. Запрещены открытые redirect-параметры, позволяющие превратить домен в open redirect.

### 6.8. Состояние объекта

- `condition_inspections` — дата, инспектор, оценка состояния, комментарий, доказательства;
- `condition_issues` — тип, severity, статус, публичность, координата/часть объекта, ответственный;
- `restoration_records` — вид работ, даты, подрядчик/организация, описание, документы, фото до/после;
- `maintenance_events` — уборка, возложение, субботник, осмотр;
- `responsible_organizations` — связь организации с объектом и период ответственности.

Статусы обращения:

`submitted → review → accepted → in_progress → resolved`  
Дополнительные: `rejected`, `duplicate`, `closed_without_action`.

Официальный осмотр и пользовательское сообщение должны различаться.

### 6.9. Народный архив

`community_contributions`:

- type: new_place/correction/photo/document/story/audio/timeline/condition_report;
- target entity;
- author public name/pseudonym;
- private contact reference;
- consent version;
- rights/license declaration;
- moderation status;
- duplicate link;
- created/updated/published timestamps.

`contribution_files`:

- MIME, size, checksum;
- quarantine/scanning status;
- private original and public derivative;
- EXIF removal status;
- attribution and license.

Статусы модерации:

`draft → submitted → screening → editor_review → changes_requested → approved → published`  
Дополнительные: `rejected`, `withdrawn`, `archived`.

### 6.10. Образовательный режим

- `educational_modules` / специальный тип маршрута;
- `quiz_questions`;
- `quiz_answers`;
- `class_sessions` с временным кодом;
- `anonymous_attempts`;
- `completion_certificates` без обязательного хранения ФИО ребёнка;
- возрастная группа и методические материалы для преподавателя.

### 6.11. Афиша

Расширить существующие события:

- связь с объектом и маршрутом;
- повторяемость;
- IANA timezone `Asia/Yekaterinburg`;
- источник и внешний ID;
- `last_synced_at`;
- статус проверки;
- регистрационная/внешняя ссылка;
- ICS export;
- автоматический архив прошедших событий;
- защита от дублей по source + external ID.

### 6.12. Организации и партнёры

- `organizations`;
- `organization_members`;
- `organization_roles`;
- владение/кураторство объектов, маршрутов и событий;
- отдельные лимиты, брендинг и аналитика;
- настройка публичного виджета;
- API keys/scopes только при появлении внешнего API.

### 6.13. База знаний для AI

- `knowledge_documents` — ссылка на конкретную опубликованную версию карточки, маршрута, человека, события или архивного материала;
- `knowledge_chunks` — фрагмент, entity/source IDs, locale, checksum, publication version;
- `knowledge_index_jobs` — статус инкрементальной индексации;
- `assistant_sessions` и `assistant_feedback` с минимальным сроком хранения;
- версия модели, prompt policy и индекса для воспроизводимости оценки.

Черновики, внутренние поля, контакты авторов и снятые с публикации версии не индексируются. Выбор полнотекстового, vector или hybrid-поиска выполняется только после аудита: не добавлять отдельную vector DB, если текущая БД или поисковый движок справляются с задачей.

## 7. Публичные функции

### F-01. Расширенное исследование карты

Добавить фильтры:

- исторический период;
- тип объекта;
- наличие аудио, видео или 3D;
- наличие архивных фотографий;
- доступность;
- связь с человеком или событием;
- наличие маршрутов;
- состояние объекта, если данные публичны.

Требования:

- существующие категории и округа сохраняются;
- фильтры отражаются в URL и доступны для шаринга;
- кнопка сброса показывает понятное состояние;
- нулевой результат предлагает ослабить фильтры;
- мобильная панель не перекрывает критические элементы карты;
- результаты доступны списком без обязательного взаимодействия с картой.

### F-02. Тематические маршруты

Публичные экраны:

1. Каталог маршрутов.
2. Карточка маршрута.
3. Предпросмотр точек, длины, времени, сложности и доступности.
4. Режим прохождения.
5. Прогресс и следующая остановка.
6. Итог маршрута и возможность поделиться.

Типовые маршруты:

- 30/60/90 минут;
- военная история;
- люди Тюмени;
- памятники центра;
- школьный;
- безбарьерный;
- вечерний;
- маршрут конкретного автора/краеведа.

GPS-режим:

- запрашивать геолокацию только после действия пользователя «Начать прогулку»;
- объяснить назначение доступа;
- не запускать аудио неожиданно;
- учитывать погрешность GPS, hysteresis и повторное вхождение в радиус;
- не проигрывать одну точку повторно без явного действия;
- предоставить ручную кнопку «Я у объекта»;
- не обещать фоновые возможности, которые браузер не гарантирует.

### F-03. Аудиопрогулка и варианты рассказа

Для объекта и остановки маршрута:

- «Коротко» — 45–60 секунд;
- «Подробно» — 3–5 минут;
- детская версия;
- версия простым языком;
- аудиоописание при наличии;
- текст каждой версии;
- следующий/предыдущий объект маршрута;
- сохранение позиции воспроизведения;
- корректное восстановление после сворачивания страницы в пределах возможностей браузера.

Генерация SpeechKit выполняется на backend. Секреты не попадают в клиент. Использовать hash текста, защиту от параллельной генерации, retry с ограничением и версионирование URL/объекта.

### F-04. Офлайн-пакет маршрута

Пользователь заранее загружает:

- данные маршрута;
- необходимые карточки;
- оптимизированные изображения;
- аудио выбранного языка;
- минимальную карту или маршрутную геометрию, если лицензия и текущий провайдер это допускают.

Требования:

- показать размер до загрузки;
- показывать прогресс и ошибку;
- позволить удалить пакет;
- версионировать manifest и инвалидировать устаревшие файлы;
- не кешировать приватные административные данные;
- иметь понятный offline fallback;
- проверить квоты Safari/Chrome/Android и graceful degradation.

### F-05. Люди, события и связанные места

Добавить:

- поиск по ФИО и вариантам написания;
- страницу человека;
- краткую биографию и хронологию;
- связанные объекты и события;
- роль человека в отношении объекта;
- архивные ссылки и источники;
- блок «Связано с этим местом» в карточке;
- блок «Другие места по теме».

Для сомнительных совпадений должна быть пометка `требует проверки`. Не объединять однофамильцев автоматически.

### F-06. «Машина времени»

- Хронология объекта.
- Архивные фото по годам.
- Сравнение «тогда/сейчас» с ползунком.
- Подпись, автор, источник и права.
- Фильтр года/периода на карте.
- Отдельная отметка приблизительной даты.
- Адаптивный режим без ползунка для screen reader: две последовательные фотографии с описаниями.

### F-07. QR/NFC около объекта

Сценарий:

1. Пользователь сканирует код.
2. Попадает прямо в карточку объекта или маршрута.
3. Без регистрации видит главный факт и кнопку аудио.
4. Выбирает язык.
5. Может продолжить к следующему месту.

В админке:

- создать кампанию;
- скачать SVG и PNG, при необходимости печатный PDF;
- проверить контраст и quiet zone;
- назначить место установки;
- выключить потерянную/повреждённую табличку;
- видеть обезличенные сканирования и конверсии в аудио/маршрут.

### F-08. Состояние и сохранение памятника

Публичный блок:

- дата последней проверки;
- обобщённое состояние;
- история реставраций;
- фото до/после;
- известные публичные работы;
- мероприятие по уходу за объектом;
- кнопка сообщения о повреждении.

Не публиковать внутренние контакты, сметы, закрытые документы и непроверенные обвинения.

### F-09. Народный архив

Формы:

- добавить старую фотографию;
- рассказать семейную историю;
- загрузить документ;
- записать/загрузить голосовое воспоминание;
- предложить новый объект;
- предложить исправление;
- сообщить о состоянии.

До отправки пользователь видит:

- что материал попадёт на модерацию;
- требования к источнику;
- варианты публичного имени;
- заявление о правах;
- согласие на обработку контакта;
- возможность отозвать материал.

Никакой пользовательский материал не публикуется напрямую.

### F-10. Школьный режим

- Маршруты по возрастным группам.
- Короткий текст и аудио.
- Вопросы на остановках.
- Код учебной сессии.
- Псевдонимное прохождение без аккаунта ребёнка.
- Агрегированный результат для преподавателя.
- Итоговый сертификат/карточка прохождения.
- Отсутствие публичного рейтинга детей.

### F-11. Доступность

- Полная клавиатурная навигация.
- Видимый focus.
- Семантические заголовки и landmarks.
- Масштабирование текста без поломки.
- Высокий контраст и поддержка системной темы.
- `prefers-reduced-motion`.
- Alt-тексты и подписи медиа.
- Транскрипты аудио и субтитры видео.
- Аудиоописания и РЖЯ-видео как отдельные варианты контента.
- Версия простым языком.
- Атрибуты безбарьерного маршрута: ступени, уклон, переходы, покрытие, места отдыха.

### F-12. Мультиязычность

- Русский остаётся основным и fallback-языком.
- Локализуются карточки, маршруты, люди, события, интерфейс и аудио.
- У каждой локали собственный editorial status.
- Машинный перевод создаёт черновик и не публикуется автоматически.
- URL содержит устойчивую локаль.
- Добавить `hreflang`, sitemap и корректные metadata.

### F-13. Умный помощник

Разрешённые сценарии:

- вопросы о текущем объекте;
- вопросы о связанных людях и событиях;
- «что посмотреть рядом за 40 минут»;
- подбор уже опубликованного маршрута;
- сборка чернового маршрута по времени, теме и доступности.

Жёсткие требования:

- использовать только опубликованную проверенную базу;
- показывать ссылки на источники и карточки;
- отвечать «в базе недостаточно данных», если подтверждения нет;
- не использовать открытый интернет по умолчанию;
- не выполнять инструкции из загруженных исторических текстов;
- не публиковать новый исторический факт;
- валидировать геометрию и доступность маршрута обычным кодом после ответа модели;
- rate limit, бюджет и feature flag;
- редактировать/обезличивать логи;
- иметь набор эталонных вопросов и регулярную оценку качества.

### F-14. Афиша

Улучшить существующий раздел:

- привязка события к объекту и маршруту;
- карточка события;
- повторяющиеся события;
- добавление в календарь через ICS;
- подписка на тему/место только после согласия;
- импорт из доверенных источников через адаптеры;
- модерация импортированных данных;
- автоматическое скрытие прошедших событий;
- защита от дублей.

### F-15. 3D и AR — отложенная фаза

Развивать уже существующие 3D-материалы:

- кликабельные точки на модели;
- пояснения материалов, символов и надписей;
- сравнение состояния до/после реставрации;
- WebAR-размещение исторического вида или утраченной детали;
- распознавание объекта камерой без распознавания лиц.

Требования:

- запрашивать камеру только после явного действия;
- capability detection;
- ручной выбор объекта как fallback;
- ограничение веса ассетов;
- отсутствие блокировки основного контента при неподдерживаемом AR.

## 8. Административные функции

### A-01. Редакционный workflow

- Очередь материалов по статусам.
- Автор, ответственный редактор и проверяющий.
- Комментарии и запрос изменений.
- Сравнение версий.
- Откат.
- Отложенная публикация.
- История действий.
- Отдельное одобрение исторических фактов, перевода и аудио.

### A-02. Индикатор готовности карточки

Админка должна показывать процент и причины:

- нет координат или адреса;
- нет обложки/alt-текста;
- нет источников;
- отсутствуют права на медиа;
- аудио устарело после изменения текста;
- не заполнена доступность;
- отсутствует краткая версия;
- возможный дубль;
- перевод не проверен.

### A-03. Производственный модуль аудио

- Раздельные тексты карточки и озвучивания.
- Короткая, полная, детская, простая и иностранная версии.
- Предпрослушивание.
- Выбор голоса и скорости.
- Словарь ударений, фамилий, топонимов и аббревиатур.
- Оценка длительности и стоимости до генерации.
- Очередь, идемпотентность, retry и отмена.
- История версий.
- Ручная загрузка профессиональной записи.
- Автоматический `stale` после изменения одобренного сценария.

### A-04. Конструктор маршрутов

- Создание точек на карте.
- Drag-and-drop порядка.
- Расчёт длины и времени.
- Проверка отсутствующих аудио/координат.
- Вступление, финал и переходы между точками.
- Предпросмотр мобильного режима.
- Дублирование маршрута как черновика.
- Публикация по расписанию.
- Создание offline manifest.

### A-05. Модерация народного архива

- Quarantine файлов.
- Автоматические проверки MIME, размера, checksum и вредоносного содержимого доступными средствами проекта.
- Поиск дублей.
- Просмотр прав/согласий.
- Запрос уточнений.
- Частичная публикация.
- Ссылка опубликованного вклада на автора и исходную заявку.
- Удаление/скрытие с сохранением audit log.

### A-06. Осмотры и работы

- Мобильная форма инспектора.
- Фото с датой; геолокация только с разрешения.
- Классификация дефектов.
- Severity и ответственный.
- Статус работ.
- История изменений.
- Публичная и внутренняя части записи.
- Отчёт по объектам/округам/организациям.

### A-07. Массовые операции

- Импорт CSV/XLSX с preview.
- Валидация до записи.
- Отчёт об ошибках по строкам.
- Поиск дублей по имени, адресу и координатам.
- Массовые теги, категории и статус.
- Пакетная генерация аудио.
- Экспорт CSV/GeoJSON.
- Идемпотентный повтор импорта.

### A-08. Кабинет партнёра

- Данные только своей организации.
- Пользователи и роли.
- Объекты, маршруты, события и QR.
- Статистика.
- Лимиты генерации и хранения.
- Экспорт отчёта.
- Настройка виджета карты.
- Отправка материалов на общую модерацию.

Необходимые задачи:

- генерация и повторная генерация аудио;
- создание короткой, детской и простой версии как редакционного черновика;
- перевод как черновик;
- генерация responsive-изображений и производных медиа;
- проверка пользовательских файлов и удаление лишнего EXIF;
- сборка offline manifest/packages;
- обновление поиска и AI-индекса после публикации;
- синхронизация и дедупликация событий;
- агрегация QR/продуктовой аналитики;
- проверка битых источников;
- напоминание об устаревших осмотрах и accessibility-данных;
- очистка данных по утверждённым retention rules.

Все jobs должны быть идемпотентными, иметь ограниченные retry/backoff, понятный failed/dead-letter status и ручной безопасный повтор. Постановка задачи после публикации должна быть согласована с транзакцией; использовать существующий outbox или добавить минимальный DB-backed outbox только при доказанной необходимости.

## 10. Аналитика

Не ограничиваться просмотрами страниц. Ввести стабильную схему событий:

- `map_open`;
- `search_submit`, `search_zero_results`;
- `filter_apply`, `filter_reset`;
- `place_open`;
- `media_view`, `model_3d_open`;
- `audio_start`, `audio_progress_25/50/75`, `audio_complete`;
- `route_view`, `route_download`, `route_start`, `route_stop_reached`, `route_complete`;
- `qr_open`;
- `favorite_add`, `visited_mark`;
- `event_view`, `calendar_add`;
- `contribution_start`, `contribution_submit`;
- `condition_report_submit`;
- `ai_question`, `ai_answer_supported`, `ai_no_answer`.

Минимальные поля:

- event name and schema version;
- timestamp;
- pseudonymous session/user ID;
- entity type and ID;
- route/campaign/organization ID;
- locale;
- device category;
- referrer category;
- coarse outcome/error code.

Не передавать в аналитику полный текст пользовательской истории, AI-вопроса, контакты, точные координаты и другие потенциально персональные данные без отдельной необходимости и согласия.

Основные воронки:

1. QR → карточка → запуск аудио → завершение.
2. Просмотр маршрута → старт → достигнутые точки → завершение.
3. Поиск → карточка → маршрут.
4. Начало вклада → отправка → одобрение → публикация.
5. Сообщение о состоянии → принято → исправлено.

## 11. Нефункциональные требования

### Производительность

- Сохранить или улучшить текущие Core Web Vitals.
- Цель LCP ≤ 2,5 секунды на p75 мобильных пользователей при нормальном 4G, если текущая архитектура позволяет.
- Не загружать 3D, архивные оригиналы и полный offline-пакет до явного действия.
- Responsive images, lazy loading, CDN/cache headers для публичных медиа.
- Кластеризация/виртуализация при росте числа объектов.
- Публичные read endpoints должны поддерживать кеширование.

### Надёжность

- Идемпотентные фоновые операции.
- Ограниченные retry с backoff.
- Dead-letter/failed status с ручным повтором; использовать существующий механизм очередей, не добавлять новый брокер без необходимости.
- Correlation ID и структурированные логи.
- Health checks для API, хранилища и внешних провайдеров.
- Backup и проверенный restore для БД и файлов.

### Безопасность

- Серверный RBAC и tenant isolation.
- CSRF-защита для cookie-based сессий.
- Санитизация rich text и пользовательского контента.
- Проверка загрузок по MIME и сигнатуре, ограничения размера.
- Signed/private upload flow, если он уже принят в проекте.
- CSP и запрет произвольных embed/script URL.
- Rate limit и anti-spam публичных форм.
- Секреты только в secret/env management, никогда в репозитории и клиентском bundle.
- Защита short links от open redirect.
- Проверка prompt injection для AI-контура.

### SEO

- Постоянные slug/URL объектов, маршрутов, людей и событий.
- Canonical URL.
- Sitemap.
- OpenGraph.
- Структурированные данные подходящих типов Schema.org.
- Индексируемый текстовый fallback для карты и карточек.

## 13. Тестовая стратегия

### Unit tests

- расчёт порядка и длительности маршрута;
- geofence/hysteresis;
- статусы workflow;
- readiness score;
- text hash и stale audio;
- short-link validation;
- повторяемость событий и timezone;
- права ролей;
- AI citation parsing and refusal;
- import validation and deduplication.

### Integration tests

- миграции на копии актуальной схемы;
- CRUD каждой сущности;
- tenant isolation;
- загрузка и модерация файлов;
- генерация аудио через mocked provider;
- QR redirect and analytics;
- offline manifest versioning;
- event import idempotency;
- public cache headers;
- AI retrieval only from published documents.

### E2E

Обязательные сценарии:

1. Открыть текущую карту и карточку существующего объекта.
2. Найти объект, добавить в избранное и отметить посещённым.
3. Запустить существующий аудиогид и открыть его текст.
4. Открыть маршрут, начать прохождение, вручную подтвердить точку и завершить.
5. Скачать и удалить offline-пакет.
6. Перейти по QR-short link.
7. Найти человека и перейти к связанному объекту.
8. Просмотреть хронологию и «тогда/сейчас».
9. Отправить вклад и убедиться, что он не опубликован до модерации.
10. Провести вклад через редакционный workflow.
11. Создать осмотр, дефект и закрыть его с фото после.
12. Пройти школьную сессию без персонального аккаунта ребёнка.
13. Добавить событие в календарь.
14. Получить AI-ответ с источниками и отказ при отсутствии данных.

### Дополнительные проверки

- автоматический accessibility scan и ручная клавиатурная проверка;
- screen reader smoke test;
- responsive visual tests на основных ширинах;
- slow 4G/performance budget;
- offline/reconnect;
- denied geolocation/camera/notification permissions;
- XSS в rich text;
- broken/oversized/wrong-MIME uploads;
- rate limit;
- backup/restore rehearsal на staging;
- rollback миграции или документированный forward-fix, если rollback небезопасен.

В тестах не обращаться к production SpeechKit/LLM и не расходовать реальные квоты, кроме отдельного защищённого smoke test.

## 14. Критерии приёмки по ключевым функциям

### Маршруты

- Редактор публикует маршрут минимум из двух валидных точек.
- Пользователь видит длину, время, точки и доступность до старта.
- Маршрут можно пройти без GPS.
- GPS не запускает одну остановку повторно.
- После перезагрузки прогресс гостя восстанавливается локально.
- Изменение маршрута создаёт новую offline-версию.

### Люди и связи

- Поиск находит ФИО и alias.
- На странице человека показаны только опубликованные связи.
- Каждая биография содержит источники или явный статус проверки.
- Однофамильцы не объединяются автоматически.

### QR/NFC

- Код остаётся рабочим после редактирования объекта.
- Redirect ведёт только на разрешённую внутреннюю цель.
- Кампания различает места установки.
- Аналитика не требует хранения точной геолокации посетителя.

### Хронология

- Поддерживаются точная, приблизительная дата и диапазон.
- Архивное фото нельзя опубликовать без источника и статуса прав.
- Сравнение доступно с клавиатуры и имеет текстовый fallback.

### Состояние

- Официальный осмотр отделён от сообщения пользователя.
- Изменение статуса записывается в audit log.
- Публичный интерфейс не показывает внутренние данные.
- Закрытие проблемы может содержать доказательство «после».

### Народный архив

- Прямая публикация невозможна.
- Файл проходит проверки до доступа редактора.
- Согласие и версия условий сохраняются.
- Автор может запросить отзыв материала.
- Публичная атрибуция отделена от контактных данных.

### AI-помощник

- Каждый исторический ответ содержит ссылки на использованные опубликованные источники.
- При отсутствии подтверждения система отказывается от догадки.
- Неопубликованный/удалённый материал не попадает в retrieval.
- Prompt injection из контента не меняет системные правила.
- Маршрут проходит обычную серверную валидацию.

## 15. Definition of Done для каждой задачи

Задача считается завершённой, если:

- выполнены функциональные критерии;
- миграция безопасна и проверена на staging-копии;
- есть unit/integration/E2E тесты пропорционально риску;

- добавлена аналитика без PII;
- UI адаптивен и доступен с клавиатуры;
- добавлены loading/empty/error/offline states;
- тексты локализуемы и не захардкожены в компонентах, если проект уже использует i18n;
- внешние провайдеры скрыты за существующим adapter/service layer;
- секреты не попали в код и клиент;
- обновлена документация;
- описаны новые env/config, миграции и rollback;
- feature flag по умолчанию безопасен;
- Codex перечислил изменённые файлы и команды проверки;
- build, lint, typecheck и тесты проходят.

## 19. Рекомендуемый первый продуктовый релиз

После завершения фундамента ближайший публичный релиз должен включать:

1. Редакторские маршруты.

4. QR/NFC-кампании с обезличенной аналитикой.

Отдельный коммерческий контур:

> паспорт объекта → редакционная проверка → аудиогид → QR → аналитика → состояние объекта → кабинет организации

Он позволяет предлагать продукт муниципалитетам, музеям, туристическим организациям и владельцам культурных объектов, не ухудшая публичную часть память.site.
